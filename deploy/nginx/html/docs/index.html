Agent API reference

Contract version 2

A durable, asynchronous human loop for agents. Submit, poll for minutes if needed, then unlock the ready result through a separate second gate.

One chain, two gates

0.10 USDCgate 1, enter queue
→
Human workpoll until ready
→
2.90 USDCgate 2, unlock result

A successful paid chain costs exactly 3.00 USDC. The request does not reach the human queue until gate 1 settles. The answer stays locked until a distinct gate 2 payment settles. Both settlements belong to the same verify_id.

Five complete free chains per wallet. One free entitlement covers both 0.10 and 2.90 USDC gates. The agent still performs submit, poll, and unlock exactly like a paid chain. A failed admitted chain grants one 0.10 USDC entry credit for the next chain, but that credit does not cover the 2.90 USDC result gate.

Required agent loop

  1. Call POST /verify. If it returns 402, pay 0.10 USDC and repeat.
  2. Store verify_id from the 202 response.
  3. Poll GET /verify/{verify_id} while status is processing.
  4. If status is failed, stop. The response reports whether an entry credit was granted.
  5. If status is ready, call POST /verify-unlock?id={verify_id}.
  6. If unlock returns 402, make the new 2.90 USDC payment and repeat the unlock.
  7. Use the answer only from a completed response.

Human work can take several minutes. Do not use a 110 second client deadline. Honor retry_after_seconds, persist the id, and resume polling later if necessary. Unanswered work expires after 60 minutes.

POST /verify-unlock?id={verify_id}

Call only after status is ready. Paid and entry-credit chains receive a new 2.90 USDC x402 requirement. Full-free chains pass without payment. A successful response is:

x402 payment

Both payments use x402 v2, scheme exact, USDC on Base mainnet eip155:8453. Sign the exact EIP-3009 authorization from the PAYMENT-REQUIRED header and repeat the same request with PAYMENT-SIGNATURE. Entry and unlock are distinct authorizations and distinct on-chain transactions.

Failure and credit

  "failure": {
    "reason": "human_timeout",
    "entry_credit_granted": true,
    "entry_credit_value_usdc": "0.10"

}

The credit is not cash. It is an auditable entitlement that pays only gate 1 of the wallet's next chain.

Callbacks, errors, and audit

A callback sends verify.ready or verify.failed. It never leaks a locked answer. Polling is authoritative. Common errors are 400 invalid input, 402 payment required, 409 wrong lifecycle action, 429 one active chain per wallet, and 503 queue full. Capacity checks happen before payment.

PostgreSQL stores the wallet, request, both list prices, both charged amounts, both transaction hashes, free uses, credits, lifecycle timestamps, and an append-only audit log. See the canonical source file docs/API.md.

MCP

The MCP endpoint is https://verifi.cloud/mcp. Its tools are verify_claim, get_verify, unlock_verify, and verifi_info. MCP covers full-free chains. Paid x402 calls use the HTTP API.
