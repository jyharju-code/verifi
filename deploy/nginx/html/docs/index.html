Agent API Reference

One endpoint, one human answer. No SDK, no account: just HTTP and the open x402 v2 payment protocol (USDC on Base).

1 · Overview

Verifi routes a claim from your agent to a real human, who accepts, rejects, or refines it. The HTTP call holds open while the human answers (up to about 110 seconds), so most verifies complete in a single request and response.

  • Free tier: 5 verifies per wallet address. Pass your address as agent_id.
  • Paid tier: $0.10 per verify, paid inline with x402. No registration. Response unlocks automatically when the payment settles.

2 · Endpoints

POST /verify

Submit a claim for human verification.

FieldTypeDescription
intentstring, max 2000What your agent is trying to do
claimstring, max 4000The claim a human should verify
agent_idstring, optionalYour wallet address (0x...). Required for the free tier.
callback_urlhttps URL, optionalWe POST the result here when the verify resolves (3 attempts). Polling always works too.

Query parameter ?wait=0..110 shortens the synchronous wait (default 110 s). Use wait=0 with callback_url in serverless environments.

Responses: 200 human answered within the wait window, 202 still pending (poll the URL below), 402 payment required, 400 invalid input.

GET /verify/{verify_id}

Poll a pending verify. Returns the same shape as POST.

{
  "verify_id": "a1b2c3d4-...",
  "status": "accepted",        // pending | accepted | rejected | refined | expired
  "verdict": "true",           // "true" | "false" | "refined" | null
  "explanation": null,         // refined free text lives here
  "response": "accepted",      // legacy field, kept for compatibility
  "response_time_ms": 28508,
  "tier": "paid",
  "unlock_paid": true,
  "created_at": "2026-07-17T01:54:35Z",
  "expires_at": "2026-07-17T02:54:35Z",
  "responded_at": "2026-07-17T01:55:03Z"
}

3 · Payment: the x402 v2 flow

After your 5 free verifies, POST /verify returns 402 Payment Required with a PAYMENT-REQUIRED response header: a base64-encoded JSON object listing accepted payment methods.

// base64-decoded PAYMENT-REQUIRED header
{
  "x402Version": 2,
  "accepts": [{
    "scheme": "exact",
    "network": "eip155:8453",                              // Base mainnet
    "amount": "100000",                                    // 0.10 USDC (6 decimals)
    "asset": "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913", // USDC on Base
    "payTo": "0x...",
    "maxTimeoutSeconds": 300
  }]
}

Your agent signs an EIP-3009 transferWithAuthorization for exactly that amount and retries the same request with the signed payload base64-encoded in a PAYMENT-SIGNATURE request header. The buyer wallet needs no ETH: the signature is off-chain and Verifi's facilitator pays the gas. On success the response carries a PAYMENT-RESPONSE header with the settlement transaction hash.

Any x402 v2 compatible client handles this handshake automatically. The official TypeScript packages are @x402/fetch and @x402/evm.

4 · Examples

Free tier (curl)

curl -X POST https://verifi.cloud/verify \
  -H "Content-Type: application/json" \
  -d '{
    "intent": "send_email",
    "claim": "The user opted in via double opt-in on 2026-07-15.",
    "agent_id": "0xYourWalletAddress"
  }'

Paid tier (Node.js, official x402 SDK)

import { wrapFetchWithPaymentFromConfig } from '@x402/fetch';
import { ExactEvmScheme } from '@x402/evm';
import { privateKeyToAccount } from 'viem/accounts';

const account = privateKeyToAccount(process.env.WALLET_PK);
const fetchPay = wrapFetchWithPaymentFromConfig(fetch, {
  schemes: [{ network: 'eip155:8453', client: new ExactEvmScheme(account) }],
});

const resp = await fetchPay('https://verifi.cloud/verify', {
  method: 'POST',
  headers: { 'content-type': 'application/json' },
  body: JSON.stringify({
    intent: 'refund decision',
    claim: 'Customer #482 qualifies for a refund under our 30-day policy',
    agent_id: account.address,
  }),
});
console.log(resp.status, await resp.json());

5 · Statuses and errors

Code / statusMeaning
200 acceptedA human confirmed the claim
200 rejectedA human rejected the claim
200 refinedA human answered with free text (in response)
202 pendingNo answer yet: poll GET /verify/{id}
expiredNo human answered within 60 minutes
400Missing or oversized intent / claim
402Free quota used and no valid PAYMENT-SIGNATURE
429One pending verify per agent_id: wait for the previous one to resolve or expire

6 · Rules and edge cases

  • One pending verify per agent_id. A new POST while your previous verify is still pending returns 429. This protects the human queue.
  • 60 minute expiry. If no human answers within 60 minutes (expires_at), the verify becomes expired. Expired free verifies still consume quota; treat expiry as "no answer".
  • unlock_paid. Free responses are always open. Paid responses unlock automatically when the x402 settlement lands. If a settlement fails after creation (unlock_paid: false), recover with POST /verify-unlock?id={verify_id}: an x402-paid endpoint ($3.00) that re-opens the response. In normal operation you never need it.
  • Expired paid verifies auto-credit. If your paid verify expires unanswered, that wallet address automatically receives one extra free verify.
  • Queue cap. When too many verifies wait for humans, POST returns 503 with Retry-After. Honor it.
  • Do not resubmit the same claim repeatedly. A human reads every request.

7 · MCP server

Verifi is also an MCP server (Streamable HTTP) at https://verifi.cloud/mcp with tools verify_claim, get_verify, and verifi_info. Add it to Claude Code, Cursor, or any MCP client and your agent can ask a human directly. The MCP surface covers the free tier; the x402-paid tier runs over plain HTTP.
